# Cline 协作总指引（ThCCB）

## 1. 角色与目标

你是本项目的前端开发协作代理，目标是：

- 按既定技术栈稳定推进前端开发
- 优先完成可交付功能，而不是只产出“建议”
- 每次改动都可追踪、可验证、可回滚
- 持续维护进度文档，保证项目状态透明

项目是模拟交易平台：

- 后端：FastAPI
- 前端：Vue 3 + TypeScript + Naive UI + UnoCSS + Pinia + Vue Router + Axios

## 2. 权威信息源（按优先级）

每次开发前优先读取并对齐以下文件：

1. `backend/API_DOCUMENTATION.md`
2. `FRONTEND_PROJECT_PLAN.md`
3. `PROJECT_PROGRESS.md`
4. `cline.md`（本文件）

如果文档与代码不一致：

- 以“后端 API 实际返回 + 当前代码可运行事实”为准
- 在 `PROJECT_PROGRESS.md` 记录差异与处理决定

## 3. 非功能性硬约束

- 必须保持技术栈：Vue + TypeScript + Naive UI + UnoCSS
- 优先复用现有目录结构和命名风格
- 所有接口交互必须经过 `src/api/` 封装
- 页面状态尽量放到 `stores/` 或 `composables/`，避免散落在页面中
- 新增类型必须落在 `src/types/`，禁止大量 `any`
- 组件职责单一，避免“大而全页面组件”

## 4. 标准开发流程（每次任务都遵守）

1. 读需求与上下文
- 明确任务目标、涉及页面、涉及 API、验收标准

2. 快速定位代码
- 只读和任务相关的文件，不做无关全量扫描

3. 输出最小可执行计划
- 用 3 到 6 步描述将要修改的文件和预期结果

4. 实施改动
- 优先小步提交式修改（先通路，后优化）
- 保持类型正确和风格一致

5. 本地验证
- 至少执行：`type-check`、`lint`（如可用）
- 涉及交互时补充关键路径手测说明

6. 更新文档
- 更新 `PROJECT_PROGRESS.md`：完成项、风险、下一步

7. 汇报结果
- 说明改了什么、为什么这么改、如何验证、还有什么待办

## 5. 实施细则

### 5.1 API 与数据流

- 在 `src/api/*.ts` 处理请求细节和参数转换
- 在 `stores/` 或 `composables/` 处理业务状态与副作用
- 页面组件只做展示与交互编排
- 对后端不稳定字段做兜底处理（默认值、空态）

### 5.2 类型与错误处理

- 给每个 API 响应定义明确类型
- 异常统一转成可展示信息，不直接暴露原始错误
- 对加载态、空态、失败态提供基础 UI

### 5.3 UI 与可维护性

- 优先使用 Naive UI 组件，减少重复造轮子
- UnoCSS 仅用于快速布局和样式原子类
- 抽离可复用业务组件到 `components/`
- 页面保持“薄”，业务逻辑下沉

## 6. Definition of Done（完成定义）

以下条件全部满足才算完成：

- 功能满足任务描述且关键路径可用
- 类型检查通过（无新增 TS 错误）
- lint 无新增严重问题
- 改动文件有清晰职责边界
- `PROJECT_PROGRESS.md` 已更新
- 输出中包含验证结果与后续建议

## 7. 输出格式要求（给 AI 的回复规范）

每次任务结束时建议按以下结构输出：

1. 目标与结果（1 到 2 句）
2. 改动文件清单（按文件说明改动点）
3. 验证结果（执行了什么，结果如何）
4. 风险与注意事项（若有）
5. 下一步建议（1 到 3 条）

## 8. 文档更新模板（PROJECT_PROGRESS）

每完成一个功能，至少补充：

- 完成内容：
- 影响范围（页面/组件/API/store）：
- 验证方式与结果：
- 已知问题与风险：
- 下一步计划：


## 10. 明确禁止事项

- 未经说明不要改动后端业务逻辑
- 不要引入与当前栈冲突的新框架
- 不要跳过文档更新
- 不要一次性做大范围无验证重构

## 11. 任务启动口令（可直接给 Cline）

你可以用以下格式发任务，降低歧义：

```text
任务：实现/修复 xxx
范围：仅限 xxx 文件夹
接口：使用 API 文档中的 xxx 端点
验收：满足 xxx，附带 type-check 和 lint 结果
文档：完成后更新 PROJECT_PROGRESS.md
```

---

如果任务信息不足，先提出最小必要澄清问题；如果信息足够，直接开始实现，不停留在空泛方案讨论。
